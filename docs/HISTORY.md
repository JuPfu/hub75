## Evolution of Pico HUB75 Drivers

### Raspberry Pi Pico HUB75 Example

The **Pico HUB75 example** demonstrates connecting a HUB75 LED matrix panel using PIO. This educational example prioritises clarity and ease of understanding.

- The colour palette is generated by modulating the **Output Enable (OE)** signal.
- **Binary Coded Modulation (BCM)** is applied row-by-row, modulating all colour bits before advancing to the next row.
- Synchronisation depends on `hub75_wait_tx_stall`.
- **No DMA is used**, leading to lower performance.

### Pimoroni HUB75 Driver

The **Pimoroni HUB75 driver** improves performance by:

- Switching from **row-wise** to **plane-wise** modulation handling.
- Using **DMA** to transfer pixel data to the PIO state machine.
- Still relying on `hub75_wait_tx_stall` for synchronisation.

## Eliminating `hub75_wait_tx_stall`

Both the **Raspberry Pi and Pimoroni** implementations use `hub75_wait_tx_stall`, which ensures:

- The state machine **stalls** on an empty TX FIFO.
- The system **waits** until the OEn pulse has finished.

However, this blocking method **prevents an efficient DMA-based approach**.

### Original `hub75_wait_tx_stall` Implementation

```c
static inline void hub75_wait_tx_stall(PIO pio, uint sm) {
    uint32_t txstall_mask = 1u << (PIO_FDEBUG_TXSTALL_LSB + sm);
    pio->fdebug = txstall_mask;
    while (!(pio->fdebug & txstall_mask)) {
        tight_loop_contents();
    }
}
```

### Alternative Approach

Instead of waiting for TX FIFO stalling, we can:

1. Modify the **PIO program** to emit a signal once the OEn pulse completes.
2. Set up a **DMA channel** to listen for this signal.
3. Establish an **interrupt handler** to trigger once the signal is received.

This approach allows fully **chained DMA execution** without CPU intervention.

## Version 2 — DMA Chains and PIO State Machines (Revised Driver)

### Overview

The following diagram illustrates the interactions between **DMA channels** and **PIO state machines** in the Version 2 revised driver:

```
[ Pixel Data DMA ] -> [ hub75_data_rgb888 PIO ]
       |
       |--> [ Dummy Pixel Data DMA ] -> [ hub75_data_rgb888 PIO ]
                  |
                  |--> [ OEn Data DMA ] -> [ hub75_row PIO ]
                           |
                           |--> [ OEn Finished DMA ] (Triggers interrupt)
```

### Step-by-Step Breakdown

1. **Pixel Data Transfer**
   Pixel data is streamed via **DMA** to the **hub75_rdata_gb888** PIO state machine, which shifts pixel data into the LED matrix.

2. **Dummy Pixel Handling**
   A secondary **dummy pixel DMA channel** adds additional pixel data to ensure correct clocking of the final piece of genuine data.

3. **OEn Pulse Generation**
   The **OEn data DMA channel** sends 32-bit words — 5-bit address information (row select) and 27-bit pulse width — to the **hub75_row** PIO state machine. This output-enable signal switches on the LEDs in the current row with the bit set in the current bitplane for the specified number of cycles.

4. **Interrupt-Driven Synchronisation**
   A final **OEn finished DMA channel** listens for the end of the pulse. An **interrupt handler** (`oen_finished_handler`) resets DMA for the next cycle.

### Version 2 Achievements

Improvements over the Pimoroni driver:

- **CPU Offloading** — Processing moved from the CPU to DMA and PIO co-processors.
- **Self-paced Pipeline** — Interlinked DMA and PIO processes run without CPU supervision.
- **No Blocking Synchronisation** — `hub75_wait_tx_stall` eliminated entirely.
- **Reduced Interrupt Complexity** — Lighter, leaner interrupt handlers.

## Version 3.0 — The Definitive Solution

Version 3.0 is a near-complete rework of the DMA/PIO pipeline. See [README.md](../README.md) for the full architectural description. The key motivation for the rework was:

- **On-demand bitplane construction** — in Version 2, bitplanes were reconstructed on the fly for every frame. Version 3.0 builds them once per `update()` / `update_bgr()` call using a dedicated DMA/PIO stage, then streams them continuously in the background.
- **Balanced Light Output** — high-weight bitplanes are split into multiple shorter segments, eliminating visible flicker at low brightness levels.
- **Per-channel CIE 1931 correction and CCM** — added as new colour accuracy features.
- **[board707's](https://github.com/board707) suggestion** of parallel loading of pixel data while BCM is running was implemented, significantly improving throughput at low-to-medium brightness settings.

For the complete Version 3.0 architecture, refresh-rate performance tables, and configuration reference, refer to [README.md](../README.md).
